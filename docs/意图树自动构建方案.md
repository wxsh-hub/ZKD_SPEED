# 意图树自动构建方案

## 一、背景与目标

### 现状问题

当前意图树通过 `IntentTreeFactory` 硬编码，只能手动运行测试类 `initFromFactory()` 灌入数据库。存在以下问题：

- 数据初始化不自动化，新建空数据库后意图分类无法工作
- 硬编码的意图节点与业务耦合，修改需要改代码重新部署
- 虽然有数据库 CRUD 接口和前端管理页面，但初始数据仍依赖硬编码

### 目标

让意图树从**知识库元数据自动构建**，硬编码作为兜底：

- 在创建/编辑知识库时声明意图元数据（领域/分类/话题/描述/示例）
- 启动时或缓存失效时，自动从所有知识库构建意图树
- 无知识库元数据时，回退到 `t_intent_node` 表或硬编码兜底

### 核心思路

每个知识库（KB）已经 1:1 对应一个 Milvus Collection，意图树的作用就是把用户问题路由到正确的 Collection。所以在 KB 上声明意图元数据，自动构建意图树，省去手动维护 `t_intent_node` 的成本。

## 二、数据库变更

文件：`resources/database/schema_table.sql`

给 `t_knowledge_base` 表新增 6 个字段：

```sql
ALTER TABLE `t_knowledge_base`
    ADD COLUMN `intent_domain`      varchar(64)  DEFAULT NULL COMMENT '意图树DOMAIN层名称，如"集团信息化"',
    ADD COLUMN `intent_category`    varchar(64)  DEFAULT NULL COMMENT '意图树CATEGORY层名称，如"人事"',
    ADD COLUMN `intent_topic`       varchar(64)  DEFAULT NULL COMMENT '意图树TOPIC层名称（可选），如"发票相关"',
    ADD COLUMN `intent_description` varchar(512) DEFAULT NULL COMMENT '语义描述，用于LLM意图匹配',
    ADD COLUMN `intent_examples`    text         DEFAULT NULL COMMENT '示例问题JSON数组',
    ADD COLUMN `intent_enabled`     tinyint(1)   DEFAULT '0' COMMENT '是否参与意图树构建 0:否 1:是',
    ADD KEY `idx_intent_enabled` (`intent_enabled`, `deleted`);
```

### 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `intent_domain` | varchar(64) | 启用时必填 | 意图树 DOMAIN 层名称，如"集团信息化"、"业务系统" |
| `intent_category` | varchar(64) | 启用时必填 | 意图树 CATEGORY 层名称，如"人事"、"OA系统" |
| `intent_topic` | varchar(64) | 可选 | 意图树 TOPIC 层名称，如"发票相关"。为空时 CATEGORY 层作为叶子节点 |
| `intent_description` | varchar(512) | 启用时必填 | 语义描述，LLM 用此字段进行意图匹配打分 |
| `intent_examples` | text | 可选 | 示例问题 JSON 数组，帮助 LLM 理解节点含义 |
| `intent_enabled` | tinyint(1) | 否 | 是否参与意图树构建，默认 0（不参与） |

### 设计决策

- `intent_enabled` 默认 0，现有 KB 不受影响，只有显式开启的 KB 才参与意图树
- `intent_topic` 可空，空时 CATEGORY 层作为叶子节点，有值时创建 TOPIC 子节点
- `intent_examples` 存储 JSON 数组字符串，格式与 `t_intent_node.examples` 一致

## 三、实现步骤

### Step 1：后端实体/DTO 扩展

修改文件：
- `knowledge/dao/entity/KnowledgeBaseDO.java` — 新增 6 个字段
- `knowledge/controller/request/KnowledgeBaseCreateRequest.java` — 新增可选意图字段
- `knowledge/controller/vo/KnowledgeBaseVO.java` — 新增意图字段供前端展示
- 新建 `knowledge/controller/request/IntentMetadataUpdateRequest.java` — 意图元数据更新请求 DTO

### Step 2：核心逻辑 — KBIntentTreeBuilder

新建文件：`rag/core/intent/KBIntentTreeBuilder.java`

这是一个 Spring `@Component`，负责从知识库元数据自动构建意图树。

#### 构建算法

```
1. 查询所有 intent_enabled=1 且 deleted=0 的 KB
2. 按 intent_domain 分组 → 创建 DOMAIN 节点
3. 每个 domain 下按 intent_category 分组 → 创建 CATEGORY 节点
4. 每个 category 下：
   - 有 intent_topic 的 KB → 创建 TOPIC 叶子节点（绑定 kbId + collectionName）
   - 无 intent_topic 的 KB → CATEGORY 自身作为叶子节点
   - 混合情况 → 无 topic 的 KB 创建名为"综合"的合成 TOPIC 节点
5. 节点 ID 用 kb-{kbId} 命名，避免与 t_intent_node 冲突
6. 始终追加 SYSTEM 节点（欢迎/问候、关于助手）
```

#### 树形结构示例

假设有以下知识库开启了意图元数据：

| KB名称 | intent_domain | intent_category | intent_topic | intent_description |
|--------|--------------|-----------------|-------------|-------------------|
| HR文档库 | 集团信息化 | 人事 | - | 招聘、入职、转正、离职、绩效、薪资等 |
| IT支持库 | 集团信息化 | IT支持 | - | VPN、邮箱、打印机、网络等IT问题 |
| 发票信息库 | 集团信息化 | 财务 | 发票相关 | 公司发票抬头、税号等开票信息 |
| OA文档库 | 业务系统 | OA系统 | 系统介绍 | OA系统整体功能说明 |
| OA安全库 | 业务系统 | OA系统 | 数据安全 | OA系统数据权限、访问控制 |

自动构建的意图树：

```
集团信息化 (DOMAIN)
├── 人事 (CATEGORY, 叶子) → HR文档库
├── IT支持 (CATEGORY, 叶子) → IT支持库
└── 财务 (CATEGORY)
    └── 发票相关 (TOPIC, 叶子) → 发票信息库

业务系统 (DOMAIN)
└── OA系统 (CATEGORY)
    ├── 系统介绍 (TOPIC, 叶子) → OA文档库
    └── 数据安全 (TOPIC, 叶子) → OA安全库

系统交互 (DOMAIN, 硬编码)
├── 欢迎与问候 (CATEGORY, 叶子)
└── 关于助手 (CATEGORY, 叶子)
```

### Step 3：修改 DefaultIntentClassifier

修改文件：`rag/core/intent/DefaultIntentClassifier.java`

修改 `loadIntentTreeFromDB()` 方法，实现**三级兜底**：

```
优先级 1：KB 元数据（intent_enabled=1）存在
  → 从 KB 构建意图树 + SYSTEM 节点

优先级 2：t_intent_node 表有数据
  → 现有 DB 加载逻辑（兼容手动管理的意图树）

优先级 3：都没有
  → IntentTreeFactory 硬编码兜底
```

### Step 4：提取 SYSTEM/MCP 节点构建方法

修改文件：`rag/core/intent/IntentTreeFactory.java`

新增静态方法：
- `buildSystemNodes()` — 返回欢迎/问候 + 关于助手节点
- `buildMCPNodes()` — 返回销售数据等 MCP 节点（如需要）

这些方法供 `KBIntentTreeBuilder` 调用，将 SYSTEM/MCP 节点追加到从 KB 构建的树中。

### Step 5：缓存失效机制

修改文件：`knowledge/service/impl/KnowledgeBaseServiceImpl.java`

- 注入 `IntentTreeCacheManager`
- 新增 `updateIntentMetadata()` 方法
- 创建/更新 KB 意图字段后清除 Redis 缓存
- 下次请求时自动从 DB 重新构建意图树

新增 API：

```
PUT /knowledge-base/{kb-id}/intent
```

修改文件：`knowledge/controller/KnowledgeBaseController.java`

### Step 6：前端页面

修改文件：
- `frontend/src/services/knowledgeService.ts` — 新增 `updateIntentMetadata` API 调用
- `frontend/src/pages/admin/knowledge/KnowledgeListPage.tsx` — 表格增加意图分类列
- 新建 `frontend/src/components/admin/IntentMetadataDialog.tsx` — 意图元数据编辑弹窗

知识库列表页显示每个 KB 的意图分类，提供"编辑意图"按钮。

### Step 7（后续优化，本次不做）：LLM 自动打标

在文档入库时，利用现有 ENHANCER 节点调用 LLM 自动分析文档内容，推荐所属的知识库分类。结果存储在 `IngestionContext.metadata` 中，管理员确认后写入 KB 的意图字段。

## 四、意图匹配机制

### 匹配流程

用户提问后，意图匹配的核心流程：

```
用户问题
  ↓
DefaultIntentClassifier.classifyTargets()
  ↓
加载意图树（Redis 缓存 → DB/KB 构建）
  ↓
提取所有叶子节点的 id/path/description/examples/type
  ↓
构造 Prompt 发送给 LLM（qwen-plus）
  ↓
LLM 返回 [{"id": "kb-123", "score": 0.92, "reason": "..."}]
  ↓
按 score 降序排序，取 score >= 阈值的 topN 节点
  ↓
根据节点 kind 路由到对应的知识库 Collection
```

### 匹配质量关键

节点的 `intent_description` 和 `intent_examples` 是匹配质量的关键：

- `description` 写得越精准，LLM 越能准确匹配
- `examples` 覆盖典型问法，帮助 LLM 理解节点含义
- 建议 description 控制在 50-200 字，examples 3-5 条

## 五、兜底机制

```
                ┌─────────────────────┐
                │  KB 元数据存在？     │
                │  (intent_enabled=1) │
                └──────────┬──────────┘
                     有 ↓        ↓ 无
              ┌──────────────┐  ┌──────────────────┐
              │ 从 KB 构建树  │  │ t_intent_node 有  │
              │ + SYSTEM 节点 │  │ 数据？            │
              └──────────────┘  └────────┬─────────┘
                                    有 ↓      ↓ 无
                             ┌──────────────┐ ┌──────────────────┐
                             │ 从 DB 加载树  │ │ IntentTreeFactory│
                             │（现有逻辑）   │ │ 硬编码兜底       │
                             └──────────────┘ └──────────────────┘
```

缓存策略：
- 意图树缓存到 Redis（key: `ragent:intent:tree`，7天过期）
- KB 意图字段变更时清除缓存
- `t_intent_node` 变更时清除缓存（现有逻辑）
- 缓存失效后下次请求自动重建

## 七、验证方式

1. 创建一个新 KB，设置 `intent_domain` / `intent_category` / `intent_description`，开启 `intent_enabled`
2. 验证意图树包含新节点（通过日志或聊天测试）
3. 发送匹配该意图的聊天问题，验证路由到正确的 Milvus Collection
4. 修改 KB 的意图字段，验证缓存清除后意图树更新
5. 关闭 `intent_enabled`，验证该 KB 不再参与意图匹配
6. 清空所有 KB 意图数据，验证回退到 `t_intent_node` 或硬编码兜底
