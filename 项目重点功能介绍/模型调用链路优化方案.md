# 模型调用链路优化方案

## 一、概述

项目采用**分层模型路由策略**：结构化任务（改写、分类、参数提取）使用低成本中等模型 `qwen-plus`，最终生成回复使用用户可选的高性能模型（默认 `qwen3-max`）。通过 `ModelSelector` 统一管理模型选择、优先级排序和故障自动降级。

---

## 二、模型配置

### 2.1 Provider 配置

文件：`bootstrap/src/main/resources/application.yaml`

项目接入三个模型服务商：

| Provider | 地址 | 用途 |
|----------|------|------|
| **bailian** | `https://dashscope.aliyuncs.com` | 阿里云百炼（通义千问系列） |
| **siliconflow** | `https://api.siliconflow.cn` | 硅基流动（GLM、Embedding） |
| **ollama** | `http://localhost:11434` | 本地模型（Qwen3-8B） |

### 2.2 聊天模型候选列表

```yaml
ai:
  chat:
    default-model: qwen3-max
    deep-thinking-model: qwen3-max
    candidates:
      - id: qwen-plus        # 百炼, qwen-plus-latest, 优先级1
      - id: qwen3-max         # 百炼, qwen3-max, 支持深度思考, 优先级3
      - id: glm-4.7           # 硅基流动, GLM-4.7, 支持深度思考, 优先级0
      - id: qwen3-local       # 本地Ollama, qwen3:8b-fp16, 优先级2
```

### 2.3 Embedding 与 Rerank 模型

| 类型 | 默认模型 | 备用模型 |
|------|----------|----------|
| Embedding | `qwen-emb-8b`（硅基流动, 4096维） | `qwen-emb-local`（本地Ollama） |
| Rerank | `qwen3-rerank`（百炼） | `rerank-noop`（空操作，优先级100） |

## 三、模型选择机制

### 3.1 ModelSelector

文件：`infra-ai/src/main/java/com/huangwei/ai/ragent/infra/model/ModelSelector.java`

选择逻辑：

```
1. 用户指定了 modelId？
   → 是：在候选列表中找到该模型作为首选
   → 否：使用 default-model（qwen3-max）作为首选

2. 深度思考模式？
   → 是：过滤掉 supportsThinking=false 的模型

3. 按 priority 升序排列候选列表

4. 首选模型提升到第一位

5. 排除熔断状态（ModelHealthStore）的模型
```

### 3.2 RoutingLLMService（故障自动降级）

文件：`infra-ai/src/main/java/com/huangwei/ai/ragent/infra/chat/RoutingLLMService.java`

`LLMService` 的主实现类，核心流程：

```
1. 调用 ModelSelector 获取候选模型列表
2. 依次尝试每个候选模型：
   - 调用对应的 ChatClient（BaiLianChatClient / SiliconFlowChatClient / OllamaChatClient）
   - 成功 → 返回结果
   - 失败（超时/错误/空内容）→ 标记该模型为不健康，尝试下一个
3. 所有模型都失败 → 抛出异常
```

### 3.3 熔断机制

通过 `ModelHealthStore` 跟踪每个模型的健康状态：
- 连续失败达到阈值 → 模型进入"open"状态，自动从候选列表中排除
- 经过恢复时间后 → 模型恢复为"closed"状态，重新加入候选列表

## 四、RAG 对话链路模型分配

### 4.1 完整调用链路

```
用户提问
  │
  ├─ [1] 问候快速匹配 ──────────────────── 零成本，静态Map匹配
  │
  ├─ [2] 查询改写/拆分 ─────────────────── qwen-plus（同步，temp=0.1）
  │
  ├─ [3] 意图分类 ──────────────────────── qwen-plus（同步，temp=0.1）
  │
  ├─ [4] 歧义检测/引导 ─────────────────── 纯规则，无LLM调用
  │
  ├─ [5] 知识库检索 + MCP工具调用 ────────── 无LLM调用
  │
  ├─ [6a] 系统回复（纯SYSTEM意图）───────── qwen-plus（流式，temp=0.7）
  │
  ├─ [6b] 外部模型兜底（检索为空）────────── qwen-plus（默认）/ 用户指定
  │
  └─ [6c] 主RAG生成回复 ────────────────── 用户指定 / qwen3-max（默认）
```

### 4.2 各环节详细参数

| 环节 | 模型 | 硬编码? | 温度 | TopP | 深度思考 | 调用方式 |
|------|------|---------|------|------|----------|----------|
| 查询改写/拆分 | `qwen-plus` | 是 | 0.1 | 0.3 | 关 | 同步 |
| 意图分类 | `qwen-plus` | 是 | 0.1 | 0.3 | 关 | 同步 |
| MCP参数提取 | `qwen-plus` | 是 | 0.1 | 0.3 | 关 | 同步 |
| 记忆摘要 | `qwen-plus` | 是 | 0.3 | 0.9 | 关 | 同步 |
| 系统回复 | `qwen-plus` | 是 | 0.7 | 0.8 | 关 | 流式 |
| 外部模型兜底 | `qwen-plus` | 部分 | 0.7 | 0.8 | 用户选择 | 流式 |
| **主RAG生成** | **用户选择** | **否** | 0/0.3 | 1/0.8 | **用户选择** | 流式 |

### 4.3 设计思路

**为什么用 `qwen-plus` 做结构化任务？**

- 查询改写、意图分类、MCP参数提取都是**结构化输出任务**（要求返回JSON或特定格式）
- 这类任务不需要创造性，`qwen-plus` 足够胜任
- 低温度（0.1）+ 低TopP（0.3）保证输出确定性
- 成本低于 `qwen3-max`，响应速度更快

**为什么用 `qwen3-max` 做最终生成？**

- 最终回复是面向用户的，需要高质量、流畅的自然语言
- `qwen3-max` 支持深度思考模式，适合复杂问题推理
- 用户可以手动选择其他模型（如 `glm-4.7`、`qwen3-local`）

## 五、文档入库链路模型分配

### 5.1 入库流水线

```
FETCHER → PARSER → ENHANCER → CHUNKER → ENRICHER → INDEXER
                        ↓                    ↓
                   文档级LLM增强          分块级LLM增强
```

### 5.2 各节点模型配置

| 节点 | 模型来源 | 说明 |
|------|----------|------|
| FETCHER | 无 | 文件获取，不调LLM |
| PARSER | 无 | Tika解析，不调LLM |
| **ENHANCER** | 流水线配置 | `EnhancerSettings.modelId`，可配可不配 |
| CHUNKER | 无 | 文本分块，不调LLM |
| **ENRICHER** | 流水线配置 | `EnricherSettings.modelId`，可配可不配 |
| INDEXER | 无 | Embedding向量化，用Embedding模型 |

- ENHANCER 支持 4 种增强任务：`CONTEXT_ENHANCE`（文本改写）、`KEYWORDS`（关键词提取）、`QUESTIONS`（问题生成）、`METADATA`（元数据提取）
- ENRICHER 支持 3 种增强任务：`KEYWORDS`、`SUMMARY`（摘要）、`METADATA`
- 两个节点都通过流水线的 `settingsJson` 配置模型，不配置时使用默认模型
